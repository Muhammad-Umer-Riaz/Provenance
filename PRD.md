# Provenance — PRD

## What We're Building

Provenance is a **schema-driven, field-level report generation engine** with a human-in-the-loop review UI. It takes a YAML template (a structured spec defining sections, fields, and per-field generation strategies) plus structured human input (a form the user fills in) and produces a formatted draft report that the user reviews, edits field-by-field, and approves before export.

The core architectural idea: **different fields in a report are generated by different strategies**. Most fields never call an LLM at all — they are boilerplate lookups, direct passthroughs of user input, or deterministic calculations. The LLM earns its place only for the ~25% of fields where prose genuinely needs to be written from structured inputs. This makes the system trustworthy, auditable, and cheap to run.

This is **not** a ChatGPT wrapper. The differentiators are:

1. Field-level orchestration — N independent field operations, each routed to its own strategy, not one big prompt → one big output
2. Structured output with JSON schema validation and per-field retries
3. Immutable audit log — every generation, regeneration, and human edit is recorded
4. Human-in-the-loop review UI with field-level diff, provenance badges, and approval gating before export
5. Eval framework measuring field-level accuracy and time-to-approval

The demo template is a **Supplier Qualification Report (SQR)** — a procurement document used across manufacturing and engineering industries to assess whether a supplier meets approved-vendor criteria. The SQR was chosen because it has rich field-level variety and contains a `classifier → narrative_llm` chain that demonstrates the most architecturally interesting orchestration pattern in the engine.

---

## Constraints

- No LLM frameworks — raw OpenAI-compatible SDK (via OpenRouter), Pydantic for structured outputs and per-field retries
- Row-Level Security on all Supabase tables — users only see their own data
- Generation is async and per-field isolated — one field failure does not abort the report
- Report cannot export until all required fields are in `approved` status
- Audit log is append-only and immutable
- YAML-only template authoring in v1
- `classifier` fields must execute before any downstream `narrative_llm` fields that reference their output
- Strategies are declared per field, not per section

## Module 1: Project Setup & Scaffold

**Build:** Repo and folder structure, Supabase project setup (Postgres + Auth + Realtime + Storage), FastAPI skeleton with health check, React + Vite + Tailwind + shadcn/ui skeleton, environment variable config (`.env` pattern), initial Supabase schema migration (templates, reports, report_fields, audit_log tables), RLS policies on all tables from day one, YAML template file loading and basic validation on startup

**Architectural concept demonstrated:** Stack cohesion — all infrastructure dependencies are wired and verified before any business logic is written. RLS is set from day one so it is never retrofitted. The YAML template is treated as a first-class input contract: loading and validating it at startup establishes the spec all other modules implement against.

---

## Module 2: YAML Template Engine + Strategy Handlers

**Build:** YAML template parser with full schema validation (sections, fields, strategies, lookup dictionaries, validation_rules), orchestrator (`orchestrator.py`) that reads a template and routes each field to its strategy handler, all 6 active v1 strategy implementations (`lookup.py`, `extractor.py`, `calculator.py`, `template_fill.py`, `classifier.py`, `narrative_llm.py`), Pydantic models for structured LLM output with per-field retry logic, LangSmith tracing on every field operation (strategy, inputs, output, latency, token cost)

**Architectural concept demonstrated:** Field-level orchestration — each field is an independent unit of work routed to exactly one strategy; the LLM operates only on the fields where prose adds genuine value (~25% of the report). The `classifier → narrative_llm` chain in the SQR (score → verdict → narrative constrained by verdict) demonstrates that field outputs can gate downstream field inputs, making orchestration order meaningful and the system's reasoning auditable.

---

## Module 3: Intake Form UI

**Build:** Auto-rendered intake form from the YAML `intake:` schema (field type → component mapping), react-hook-form + zod validation with schemas derived directly from YAML field definitions, TanStack Table for the scorecard, risk register, and CAR table inputs, multi-select component for certifications, form submission to POST /reports, basic `/templates` browse page and `/reports/new` route

**Architectural concept demonstrated:** Schema-driven UI — the YAML template is the single source of truth for both the generation engine and the intake form. No form fields are hand-coded; add a field to the YAML and it renders in the form automatically. Form validation constraints are derived from the same YAML definitions used by the backend validator, eliminating drift between client and server validation rules.

---

## Module 4: Generation Pipeline + Audit Log

**Build:** POST /reports/:id/generate (async background task), per-field execution loop with isolated failure handling and retry (one failed field does not abort the report), Supabase Realtime events emitted on each field completion (field_id, status, value preview), `validator.py` running all template-defined `validation_rules` post-generation, append-only `audit_log.py` recording every generation event (field_id, strategy, inputs snapshot, output, timestamp, model if applicable), `/reports/:id/generate` progress UI page with live field-by-field status

**Architectural concept demonstrated:** Async resumable generation with immutable provenance. The per-field isolation pattern means the report is not a single atomic operation — it is N independent tasks that can fail and retry independently. The audit log is the trust mechanism in regulated environments: every value in the final report is traceable to the exact inputs, strategy, and (where applicable) model invocation that produced it.

---

## Module 5: Review UI

**Build:** Three-pane review layout (left pane: intake data summary / centre pane: draft report with field status badges / right pane: field metadata panel on hover), field status badges (draft / edited / approved) with colour coding, strategy tags visible on hover in centre pane, inline field edit with before/after diff display, per-field regeneration button (POST /reports/:id/fields/:field_id/regenerate), export gate logic (export button enabled only when all required fields reach `approved` status), PATCH /reports/:id/fields/:field_id for user edits (writes to audit log)

**Architectural concept demonstrated:** Human-in-the-loop as a first-class architectural requirement. The review UI makes the field-level orchestration visible and auditable — the engineer can see which strategy generated each value, what inputs were used, and whether the output was accepted or overridden. The approval gate is the enforcement mechanism: no draft content can leave the system. This is the differentiator between Provenance and a system that just generates and dumps text.

---

## Module 6: Export

**Build:** POST /reports/:id/export endpoint with format parameter (pdf / docx / json), Playwright/Chromium PDF renderer (renders the same HTML as the review UI centre pane — pixel-perfect match to what was approved), python-docx DOCX renderer with section and table mapping, JSON export (full report state: field values, strategies used, approval status, full audit trail), export UI page with format selector and file download, export blocked by backend if any required field is not `approved`

**Architectural concept demonstrated:** Multiple export targets from a single approved report state. The JSON export is not just a convenience download — it exposes the full report object graph (fields, strategies, audit trail) for downstream consumption, archiving, or API integration. Playwright is chosen over WeasyPrint so the PDF is a guaranteed pixel-perfect match to the review UI — there is no separate PDF template to maintain and no divergence between what was approved and what was exported.

---

## Module 7: Eval Framework

**Build:** Synthetic SQR test set covering all 3 verdict outcomes (Approved / Conditional / Rejected), all risk tiers, and validation edge cases (Conditional verdict with no CARs, breach-level performance with Approved verdict), `eval/field_accuracy.py` — measures per-field strategy output against expected values (deterministic fields exact-match; `narrative_llm` fields scored by LLM judge on groundedness and factual consistency), `eval/time_to_approval.py` — tracks regeneration count and human edit count per field before approval, results stored as versioned JSON in `eval/results/`

**Architectural concept demonstrated:** Field-level evaluation mirrors field-level generation. Because each strategy is an independent unit, it can be measured independently — `calculator` and `classifier` fields can be tested deterministically without any LLM calls; `narrative_llm` fields require an LLM judge. Time-to-approval is the production success metric for the review UI: a well-tuned prompt generates a narrative the engineer approves on the first pass without regeneration.

---

## Module 8: Additional Templates

**Build:** NCR (Non-Conformance Report) YAML template skeleton with field-level strategy assignments (different `classifier` chain: defect severity → escalation path), Site Acceptance Test (SAT) YAML template skeleton (demonstrates `calculator` fields for test pass/fail thresholds), `docs/template-authoring.md` guide explaining the YAML schema, strategy selection decision rules, and lookup dictionary structure

**Architectural concept demonstrated:** Engine generalisation — three templates with different field mixes and different `classifier` chains prove this is a platform, not a single-purpose SQR tool. The YAML authoring guide is the handoff artefact: an engineer who understands the strategy taxonomy can build a new report template without modifying any Python code.

---

## Module 9: Production Deployment & Polish

**Build:** Dockerfile for FastAPI backend, React production build → S3 bucket → CloudFront distribution (HTTPS, global CDN), EC2 t3.small (Docker Compose + Nginx reverse proxy + HTTPS via Certbot), GitHub Actions CI/CD pipeline (push to main → build Docker image → push to ECR → SSH deploy to EC2), rate limiting middleware (slowapi — per-IP limits on generate and export endpoints), README.md, DECISIONS.md fully populated, demo report walkthrough recorded

**Architectural concept demonstrated:** Infrastructure as code — the entire production environment is described in two files (`docker-compose.yml` + Dockerfiles), version-controlled alongside the application code, and readable by any engineer without hidden platform configuration. The deployment pattern mirrors DocChat intentionally: cognitive efficiency from reusing known infrastructure; the engineering effort stays in the application layer, not the ops layer.
