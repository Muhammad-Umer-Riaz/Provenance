# Module 7: Eval Framework — Plan

## Context

Module 7 adds an offline evaluation layer that validates the generation engine before
production deployment. The original plan was drafted early; this revision reflects what
the codebase actually looks like after Modules 1–6 are complete.

Three things the module proves:
1. **Field correctness** — deterministic fields exact-match expected values across all
   verdict/risk/qualification combinations.
2. **Narrative quality** — LLM-written fields score above a minimum groundedness/consistency
   threshold. Below that threshold the script exits non-zero (usable as a gate).
3. **Production effort** — audit log shows how many regenerations and edits each field
   needs in a real report before approval.

---

## Codebase facts that changed the original plan

| Finding | Impact |
|---|---|
| `validator.py` is called separately in `generation.py` after `orchestrator.generate()` returns — it's a plain function `run_validation_rules(template, context)` | We CAN call the validator from `field_accuracy.py` — add `expected_warnings` to test cases |
| PATCH on failed fields: backend allows any value edit regardless of status; only `approved` action is blocked for `failed`/`pending` | The failed-field dead end is a frontend-only gap — backend already supports manual fill-in |
| `narrative_model = "anthropic/claude-haiku-4-5"` (exact string in config.py) | Judge model confirmed; no change needed |
| Judge prompt used `ctx.composite_score` — `ctx` is a plain dict | Bug: must use `ctx["composite_score"]` notation |
| `exported` is NOT a separate `event_type` — export is identified by `field_id="_export"` sentinel only | Drop the claim; query filter `.neq("field_id", "_export")` is correct and sufficient |
| Weights are decimals (0.20 = 20%) summing to 1.0 | composite_score is in 1–5 range; verdict thresholds are correct |

## T0 — Frontend fix: failed-field edit input

**File:** `frontend/src/pages/ReportReview.tsx`

`isLocked` previously included `'failed'`, which blocked both `canApprove` (correct) and
`canEdit` (wrong). Split into:
- `canApprove = !['failed', 'pending', 'generating'].includes(field.status)` — unchanged
- `canEdit = !['pending', 'generating'].includes(field.status) && !isTable` — `failed` removed

After a user edits a failed field it becomes `edited`, then it can be approved normally.

## T1 — Test set: `eval/test_sets/sqr_cases.json`

15 cases. Each follows the `sample_intake` shape from `conftest.py` with `risk_category`
added to each `risk_register` row (per Decision 21).

### Fields in `expected` (7 deterministic)
- `composite_score` (float, ±0.001 tolerance) — weights are decimals summing to 1.0
- `qualification_verdict` (string: Preferred/Conditional/Probationary/Rejected)
- `overall_risk_tier` (string: High/Medium/Low)
- `otd_sla_status` (string: Pass/Watch/Breach)
- `defect_sla_status` (string: Pass/Watch/Breach)
- `breach_count` (int, exact)
- `overdue_car_count` (int, exact)

`next_review_date` is excluded — date-dependent.

### Threshold reminders (used to derive expected values)
- Verdict: Preferred (≥4.0), Conditional (≥3.5), Probationary (≥2.5), Rejected (<2.5)
- Risk tier: High (max priority ≥15), Medium (≥8), Low (<8); priority = likelihood × impact
- OTD: Pass (≥90%), Watch (≥85%), Breach (<85%)
- Defect: Pass (≤1.5%), Watch (≤3.0%), Breach (>3.0%)
- Invoice: Pass (≥95%), Watch (≥90%), Breach (<90%)
- NCR count: Pass (≤2), Watch (≤4), Breach (>4)
- NCR close: Pass (≤14d), Watch (≤21d), Breach (>21d)

## T2 — `eval/field_accuracy.py`

```
python eval/field_accuracy.py [--skip-llm] [--case CASE_ID]
```

**Prerequisites:** backend venv active; `OPENROUTER_API_KEY` in `backend/.env`.

- Loads template YAML → `TemplateSchema`
- For each case: runs `ReportOrchestrator.generate()` (mocked or live)
  - Mock target: `"app.orchestrator.execute_narrative_llm"`
- Exact-match check on 7 deterministic fields (float tolerance ±0.001)
- Calls `run_validation_rules(template, ctx)` and compares `fired_ids` vs `expected_warnings`
  if present in the case (optional check)
- LLM judge (when not `--skip-llm`): calls OpenRouter `anthropic/claude-haiku-4-5` for each
  of 6 narrative fields; skips `car_summary` when `corrective_actions` is empty
- Hard threshold: mean groundedness < 3.5 → `sys.exit(1)`
- Deterministic pass rate < 90% → `sys.exit(1)`
- Output: `eval/results/field_accuracy_{YYYYMMDD_HHMMSS}.json`

## T3 — `eval/time_to_approval.py`

```
python eval/time_to_approval.py --report-id <UUID>
```

- Loads `backend/.env` before importing `app.database`
- Queries `audit_log` for the given report, excluding `field_id="_export"` rows
- Groups by field_id, counts: generated, regenerated, edited, approved
- Computes: `total_generations`, `first_pass` (no regen, no edit, approved)
- Aggregate: `first_pass_rate`, `mean_regenerations_per_field`, `mean_edits_per_field`
- Output: `eval/results/time_to_approval_{report_id[:8]}_{YYYYMMDD_HHMMSS}.json`

---

## T4 — `eval/compare_runs.py`

```
python eval/compare_runs.py <run1.json> <run2.json>
```

Pure stdlib. Diffs two `field_accuracy` result JSONs:
- Deterministic pass rate delta; shows fixed/new failures
- Validation warnings pass rate delta
- Mean groundedness and factual_consistency deltas
- Per-field groundedness delta, sorted by biggest gain
- Total elapsed time delta
