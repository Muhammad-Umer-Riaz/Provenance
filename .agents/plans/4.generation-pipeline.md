# Module 4: Generation Pipeline + Audit Log

## Context

Module 4 is the core execution layer of Provenance. It connects the intake wizard (Module 3) to the review UI (Module 5) by running the field-level generation pipeline asynchronously, streaming per-field status via Supabase Realtime, and writing an immutable audit log for every field event. Without this module the report is just stored intake data; after it the report is a generated draft ready for human review.

**Key decisions confirmed for this module:**
- Async mechanism: FastAPI `BackgroundTasks` (no external queue — v1 scope)
- Realtime: Supabase Postgres Changes on `report_fields` (auto-fires on DB inserts/updates — no manual broadcast needed)
- Score/verdict: denormalized onto `reports` table via new migration (fast list-page queries, no JOIN)
- Validation warnings: stored as JSONB on `reports` table in same migration
- Audit inputs_snapshot: resolved inputs only — strategy config + rendered prompt/expression, not full intake_data
- Progress UI: grouped by section, expandable, auto-redirects to `/reports/:id/review` on completion
- Pipeline trigger: generate page calls `POST /generate` on mount (wizard navigates there; page owns the trigger)

---

## Architecture

```
Wizard submit
  └─ PATCH /api/reports/:id (intake_data saved, status unchanged)
  └─ navigate to /reports/:id/generate

GeneratePage mount
  └─ GET /api/reports/:id          (load report metadata for UI)
  └─ GET /api/reports/:id/fields   (load initial pending rows for display)
  └─ POST /api/reports/:id/generate  ── returns 202 immediately
  └─ subscribe to Realtime (postgres_changes on report_fields where report_id=eq.:id)

Background task (_run_generation)
  1. Pre-insert all fields as status=pending in report_fields (triggers Realtime for initial list)
  2. Run orchestrator with on_field_start / on_field_complete callbacks
     ├─ on_field_start: UPDATE report_fields status=generating (Realtime fires → spinner in UI)
     └─ on_field_complete: UPSERT report_fields status=draft|failed + write audit_log entry
  3. Run validator.py (template validation_rules against final context)
  4. Extract score + verdict from known SQR field IDs
  5. UPDATE reports SET status=review, score=…, verdict=…, validation_warnings=…
     └─ Frontend detects report status=review → auto-navigate to /reports/:id/review
```

### Step 2 — audit_log.py
**File:** `backend/app/audit_log.py`

Single async function used by the generation pipeline:

```python
async def log_field_event(
    db,           # supabase service client
    report_id: str,
    field_id: str,
    event_type: str,        # generated | regenerated | edited | approved | failed
    strategy: str,
    inputs_snapshot: dict,  # strategy-specific, see below
    output_value: str | None,
    model: str | None = None,
) -> None
```

`inputs_snapshot` per strategy:
- `lookup`: `{source: field.source}`
- `extractor` (fields): `{fields: field.fields}`
- `extractor` (table): `{source: field.source, computed_columns: [c.name ...]}`
- `calculator`: `{expression: field.expression}`
- `template_fill`: `{template: field.template}`
- `classifier` (rules): `{input: field.input, rules_count: N}`
- `classifier` (LLM): `{input: field.input, model: config.classifier_model}`
- `narrative_llm`: `{prompt_template: field.prompt[:500], model: config.narrative_model}`
- `direct_input`: `{field_id: field.id}`

Uses service key client (bypasses RLS — audit_log is service-write-only by policy design from Module 1).

**Validation:** call function with mock args, confirm row appears in audit_log via Supabase MCP.

### Step 4 — Orchestrator changes
**File:** `backend/app/orchestrator.py`

Add two optional async callbacks to `generate()`:

```python
async def generate(
    self,
    on_field_start: Callable | None = None,
    on_field_complete: Callable | None = None,
) -> dict:
    for section in self.template.sections:
        for field in section.content:
            if on_field_start:
                await on_field_start(field, section)
            result, error = await self._process_field_safe(field)
            if on_field_complete:
                await on_field_complete(field, section, result, error)
    return self.context
```

Add `_process_field_safe` wrapper:

```python

        return result, None
    except Exception as e:
        self.context[field.id] = field.null_value
        return None, e
```

### Step 5 — Generation route + background task
**File:** `backend/app/routes/generation.py`

#### Endpoint

```
POST /api/reports/{report_id}/generate
```

- Auth: extract user_id from Bearer token (same pattern as reports.py)
- Load report from DB, verify ownership
- State machine:
  - `draft` or `generating` → proceed (idempotent on refresh)
  - `review` or `approved` → clear existing report_fields, reset to generating, restart
  - `exported` → 400 (cannot regenerate an exported report)
- Transition report status to `generating`
- Add `BackgroundTasks` task: `_run_generation(report_id, user_id)`
- Return `202 Accepted` with `{status: "generating", report_id: "..."}`

#### Background task

```python
async def _run_generation(report_id: str, user_id: str) -> None:
```

1. **Load**: fetch report from DB → get template from `get_loaded_templates()`
2. **Pre-insert pending rows**: for every field in every section, upsert `report_fields` with `status=pending` (triggers Realtime so UI shows full field list immediately)
3. **Orchestrate**:

   ```
   on_field_start(field, section):
       UPDATE report_fields SET status=generating WHERE report_id=… AND field_id=…

   on_field_complete(field, section, result, error):
       status = 'failed' if error else 'draft'
       value  = str(result) if result is not None else None
       UPSERT report_fields (status, value)
       log_field_event(…, inputs_snapshot=_build_snapshot(field), output_value=value, model=…)
   ```

4. **Validate**: `warnings = run_validation_rules(template, context)`
5. **Extract score/verdict**: read from `context` using known SQR field IDs (identify from YAML during implementation — look for the `calculator` field producing the composite score and the `classifier` field producing the audit verdict)
6. **Finalize**: `UPDATE reports SET status=review, score=…, verdict=…, validation_warnings=…`
7. **Catastrophic failure** (uncaught exception): `UPDATE reports SET status=generating` left as-is (surface to UI as stalled) — log the error server-side. Acceptable for v1.

Helper `_build_snapshot(field)`: returns strategy-specific dict (as specified in Step 2).

**Validation:** POST /generate on a draft SQR report; confirm all report_fields rows created; confirm audit_log populated; confirm reports row updated to review with score/verdict.

### Step 10 — Realtime subscription helper
**File:** `frontend/src/lib/realtime.ts` (new)

```typescript
import { supabase } from './supabase'
import type { ReportField } from '../types'

export function subscribeToReportFields(
  reportId: string,
  onFieldUpdate: (field: ReportField) => void,
): ReturnType<typeof supabase.channel> {
  return supabase
    .channel(`report_fields:${reportId}`)
    .on(
      'postgres_changes',
      { event: '*', schema: 'public', table: 'report_fields', filter: `report_id=eq.${reportId}` },
      (payload) => onFieldUpdate(payload.new as ReportField),
    )
    .subscribe()
}
```

Realtime fires via Postgres Changes (REPLICA IDENTITY FULL set in migration). RLS on report_fields ensures users only receive their own rows.

### Step 11 — GeneratePage.tsx
**File:** `frontend/src/pages/GeneratePage.tsx`

#### State
```typescript
report: ReportResponse | null
fieldsBySection: Map<string, ReportField[]>  // section_id → fields
totalFields: number
completedFields: number  // draft + failed + approved
```

#### On mount (in order)
1. `getReport(id)` — populate header (supplier name, template)
2. `getReportFields(id)` — seed fieldsBySection with initial state (all pending, or in-progress if refreshed)
3. `generateReport(id)` — POST /generate (202 returned quickly; task runs in background)
4. Subscribe via `subscribeToReportFields(id, handleFieldUpdate)` — update state on each Realtime push

#### Completion detection
On every Realtime update: if `report.status === 'review'` OR if all fields are `draft`/`failed`/`approved`:
- Unsubscribe Realtime channel
- `navigate('/reports/${id}/review')`

Subscribe to `reports` table changes as well (to detect status=review): same pattern, filter `id=eq.:id`.

#### UI layout

```
[Header]  Generating report — {supplier_name}    {N}/{total} fields
[Progress bar — proportional fill]

For each section (declaration order):
  ┌─────────────────────────────────────────────────────┐
  │  § 01 · Supplier & Engagement Header    [4/6 ✓]  ▼ │
  │  ┌────────────────────────────────────────────────┐ │  ← expanded
  │  │  ○ supplier_name      draft                    │ │
  │  │  ⟳ composite_score   generating               │ │
  │  │  ○ audit_verdict      pending                  │ │
  │  └────────────────────────────────────────────────┘ │
  └─────────────────────────────────────────────────────┘

[Footer]  "Generation typically takes 30–60 seconds."
```

Status icons: `○` pending · `⟳` generating (animated) · `✓` draft · `✗` failed
