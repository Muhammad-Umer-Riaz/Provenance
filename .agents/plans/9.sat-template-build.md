## Step 1 — Read and validate schema constraints

Before writing the YAML, verify which fields FieldSchema accepts. Key constraints from `schemas.py`:
- `strategy` must be one of the declared literals
- `computed_columns` only valid on extractor fields
- `condition` is a string expression evaluated by `evaluate_expression()`
- `source` maps to an intake field id for extractor tables
- `rules` is a list of `{condition, output}` for classifier fields
- `prompt`, `exemplars`, `max_tokens` only on narrative_llm fields

### Sections

```yaml
sections:

  - id: equipment_header
    title: "Equipment & Site Header"
    order: 1
    content:

      - id: report_header_line
        type: structured_block
        strategy: template_fill
        template: "SAT-{{report_id}} · {{intake.equipment_model}} · {{intake.site_name}}"

      - id: equipment_details
        type: structured_block
        strategy: extractor
        fields:
          - equipment_model
          - equipment_serial
          - equipment_category
          - site_name
          - site_location
          - customer_name
          - customer_contact_name
          - customer_contact_title
          - installation_date
          - test_date
          - evaluator_name
          - acceptance_standard

      - id: issue_date
        type: field
        strategy: calculator
        expression: "today"
        format: "YYYY-MM-DD"
        label: "Issue date"

      - id: acceptance_scope
        type: narrative
        strategy: lookup
        source: "acceptance_scope_statements[equipment_category]"

  - id: test_conditions
    title: "Test Conditions"
    order: 2
    content:

      - id: test_conditions_table
        type: table
        strategy: extractor
        source: test_conditions

      - id: conditions_narrative
        type: narrative
        strategy: narrative_llm
        prompt: |
          Write 1–2 sentences on ambient test conditions at time of acceptance test.
          Ambient temp: {{test_conditions[0].ambient_temp_c}}°C
          Relative humidity: {{test_conditions[0].humidity_pct}}%
          Line voltage: {{test_conditions[0].line_voltage_v}}V
          Coolant supply: {{test_conditions[0].coolant_temp_c}}°C
          Compressed air: {{test_conditions[0].compressed_air_bar}} bar
          Note if conditions were within acceptable ranges (temp 15–30°C, humidity 20–70%).
          Omit any field that shows "—" — it was not recorded.
          Style: factual, third-person, technical.
        max_tokens: 100

  - id: performance_tests
    title: "Performance Test Results"
    order: 3
    content:

      - id: test_results_table
        type: table
        strategy: extractor
        source: test_results
        computed_columns:
          - { name: within_spec,
              expression: "'Pass' if (measured_value >= spec_min and measured_value <= spec_max) else 'Fail'",
              label: "Within spec?" }

      - id: pass_count
        type: field
        strategy: calculator
        expression: "count(test_results_table, within_spec='Pass')"
        label: "Tests passed"

      - id: fail_count
        type: field
        strategy: calculator
        expression: "count(test_results_table, within_spec='Fail')"
        label: "Tests failed"

      - id: total_tests
        type: field
        strategy: calculator
        expression: "len(test_results)"
        label: "Total tests"

      - id: effective_conditional_limit
        type: field
        strategy: calculator
        expression: "conditional_fail_limit or 1"
        label: "Conditional fail limit"

      - id: test_verdict
        type: field
        strategy: classifier
        input: fail_count
        rules:
          - { condition: "fail_count == 0",                                    output: "Pass" }
          - { condition: "fail_count <= effective_conditional_limit",          output: "Conditional" }
          - { condition: "fail_count > effective_conditional_limit",           output: "Fail" }

      - id: results_narrative
        type: narrative
        strategy: narrative_llm
        prompt: |
          Write a 2–3 sentence analytical summary of the performance test results.
          Overall: {{pass_count}} passed, {{fail_count}} failed of {{total_tests}} total tests. Verdict: {{test_verdict}}.

          Test results (reference specific test IDs and measured values in the narrative):
          1. {{test_results_table[0].test_id}}: {{test_results_table[0].parameter}} — measured {{test_results_table[0].measured_value}} {{test_results_table[0].unit}} (spec {{test_results_table[0].spec_min}}–{{test_results_table[0].spec_max}}) → {{test_results_table[0].within_spec}}
          2. {{test_results_table[1].test_id}}: {{test_results_table[1].parameter}} — measured {{test_results_table[1].measured_value}} {{test_results_table[1].unit}} (spec {{test_results_table[1].spec_min}}–{{test_results_table[1].spec_max}}) → {{test_results_table[1].within_spec}}
          3. {{test_results_table[2].test_id}}: {{test_results_table[2].parameter}} — measured {{test_results_table[2].measured_value}} {{test_results_table[2].unit}} (spec {{test_results_table[2].spec_min}}–{{test_results_table[2].spec_max}}) → {{test_results_table[2].within_spec}}
          4. {{test_results_table[3].test_id}}: {{test_results_table[3].parameter}} — measured {{test_results_table[3].measured_value}} {{test_results_table[3].unit}} → {{test_results_table[3].within_spec}}
          5. {{test_results_table[4].test_id}}: {{test_results_table[4].parameter}} — measured {{test_results_table[4].measured_value}} {{test_results_table[4].unit}} → {{test_results_table[4].within_spec}}
          (Skip any row where test_id shows "—")
          Lead with the overall pass/fail count. Name any failed tests specifically with their measured vs spec values.
          Rules: Only reference test IDs and values from the table above. Do not infer causes of failure.
          Style: factual, third-person, engineering register.
        exemplars:
          - "All 5 parametric tests passed within specification, with [test_id] returning the tightest margin at [value] [unit] against a spec of [min]–[max].
             The equipment met acceptance criteria without deviation under [acceptance_standard].
             Test verdict: Pass."
          - "[fail_count] of [total] tests failed specification. [test_id] measured [value] [unit] against a maximum specification of [max] [unit], exceeding the limit by [delta].
             Remaining [pass_count] tests passed within tolerance.
             Verdict: Conditional — resolution of the identified deviation is required before handover."
        max_tokens: 200

  - id: measurement_analysis
    title: "Measurement Analysis"
    order: 4
    content:

      - id: measurement_data
        type: structured_block
        strategy: extractor
        fields:
          - measurement_type
          - measurement_headline_value
          - measurement_headline_unit
          - measurement_secondary_value
          - measurement_secondary_unit

      - id: engineer_note
        type: narrative
        strategy: extractor
        fields:
          - engineer_observation

      - id: figure_caption
        type: structured_block
        strategy: template_fill
        template: "Figure: {{intake.measurement_type}} — {{intake.measurement_headline_value}} {{intake.measurement_headline_unit}}"

      - id: measurement_narrative
        type: narrative
        strategy: narrative_llm
        prompt: |
          Write a 2–3 sentence technical paragraph describing the key measurement result.
          Measurement type: {{intake.measurement_type}}
          Headline value: {{intake.measurement_headline_value}} {{intake.measurement_headline_unit}}
          Secondary value: {{intake.measurement_secondary_value}} {{intake.measurement_secondary_unit}} (omit if blank or "—")
          Engineer observation: {{engineer_note}}

          State the measurement type and headline value in the first sentence.
          Incorporate the engineer's observation exactly as stated — do not add interpretations beyond what is written.
          If a secondary value is present, include it in the second sentence.
          Note: the figure itself is referenced but not reproduced here; the narrative is grounded in the structured values and engineer annotation above.
          Style: factual, third-person, technical.
        exemplars:
          - "The [measurement_type] recorded a headline value of [value] [unit], [secondary value if present].
             [Engineer observation verbatim or closely paraphrased].
             This result [aligns with / falls outside] the expected performance envelope for this equipment category."
        max_tokens: 150

  - id: deviations
    title: "Deviations & Corrective Actions"
    order: 5
    content:

      - id: deviation_count
        type: field
        strategy: calculator
        expression: "len(deviations) if deviations else 0"
        label: "Deviation count"

      - id: acceptance_conditions
        type: narrative
        strategy: lookup
        source: "acceptance_conditions_blocks[test_verdict]"
        condition: "test_verdict != 'Pass'"

      - id: deviations_table
        type: table
        strategy: extractor
        source: deviations
        condition: "deviation_count > 0"

      - id: deviation_summary
        type: narrative
        strategy: narrative_llm
        prompt: |
          Write a 2–3 sentence deviation summary.
          Total deviations: {{deviation_count}} · Verdict: {{test_verdict}}

          Deviations (reference specific test IDs and severities):
          DEV-1: {{deviations[0].test_id}} — {{deviations[0].description}} (severity: {{deviations[0].severity}}, action: {{deviations[0].action_required}}, owner: {{deviations[0].owner}}, due: {{deviations[0].due_date}})
          DEV-2: {{deviations[1].test_id}} — {{deviations[1].description}} (severity: {{deviations[1].severity}}, action: {{deviations[1].action_required}}, owner: {{deviations[1].owner}}, due: {{deviations[1].due_date}})
          DEV-3: {{deviations[2].test_id}} — {{deviations[2].description}} (severity: {{deviations[2].severity}}, action: {{deviations[2].action_required}}, owner: {{deviations[2].owner}}, due: {{deviations[2].due_date}})
          (Skip any line where test_id shows "—")
          Name each deviation by test_id and severity. State the required action and owner for Critical and Major items.
          Do not add root cause analysis beyond what is stated in the description field.
          Style: factual, third-person, engineering register.
        exemplars:
          - "[deviation_count] deviation(s) were identified during acceptance testing. [test_id] ([severity]): [description] — [action_required], assigned to [owner] with due date [date].
             [Second deviation if present, else omit sentence].
             All deviations must be resolved and verified before equipment is released for production use."
        max_tokens: 180
        condition: "deviation_count > 0"

  - id: acceptance_statement
    title: "Acceptance Statement"
    order: 6
    content:

      - id: tests_to_close
        type: field
        strategy: calculator
        expression: "fail_count"
        label: "Tests requiring closure"

      - id: executive_summary
        type: narrative
        strategy: narrative_llm
        prompt: |
          Write a 2–3 sentence executive summary of the acceptance test outcome.
          Equipment: {{intake.equipment_model}} (serial: {{intake.equipment_serial}})
          Site: {{intake.site_name}}, {{intake.site_location}}
          Customer: {{intake.customer_name}}
          Acceptance standard: {{intake.acceptance_standard}}
          Test verdict: {{test_verdict}}
          Tests: {{pass_count}}/{{total_tests}} passed
          Deviations logged: {{deviation_count}}
          State the verdict clearly in the first sentence.
          If deviation_count > 0, note that open deviations must be resolved per the conditions block.
          If test_verdict is Pass, note readiness for handover to customer operations.
          Style: factual, third-person, professional engineering language.
        exemplars:
          - "The [equipment_model] installation at [site_name] has [passed / received conditional acceptance / failed] acceptance testing under [acceptance_standard].
             [Pass: Equipment is released for customer operations. | Conditional/Fail: [deviation_count] deviation(s) require resolution before handover.]
             This report was prepared by [evaluator_name] on [issue_date]."
        max_tokens: 180

      - id: signatory_block
        type: narrative
        strategy: lookup
        source: signatory_block
```

### SLA thresholds and validation rules

```yaml
sat_thresholds:
  ambient_temp_c:  { min: 15, max: 30 }
  humidity_pct:    { min: 20, max: 70 }

validation_rules:

  - id: spec_completeness
    description: "All test result rows must have both spec_min and spec_max defined"
    check: "all([r['spec_min'] is not None and r['spec_max'] is not None for r in test_results])"
    severity: error

  - id: total_tests_minimum
    description: "At least 3 performance tests required for a valid acceptance"
    check: "len(test_results) >= 3"
    severity: error

  - id: deviation_verdict_alignment
    description: "Fail verdict should have at least one deviation logged"
    check: "test_verdict != 'Fail' or len(deviations) >= 1"
    severity: warning
    message: "Verdict is Fail but no deviations have been entered. Add deviations for all failed tests."

  - id: conditional_needs_dates
    description: "Conditional verdict — all deviations must have due dates"
    check: "test_verdict != 'Conditional' or all([d.get('due_date') for d in deviations])"
    severity: warning
    message: "Verdict is Conditional but one or more deviations are missing a due date."
```
