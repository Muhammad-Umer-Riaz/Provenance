# SAT Template ‚Äî End-to-End Engineering Spec

**Template ID:** `site-acceptance-test`  
**Version:** 1.0  
**Domain:** Equipment commissioning / capital equipment installation  
**Status:** Production-ready YAML written; awaiting full engine integration

---

## 1. Context

### What SAT reports are

A Site Acceptance Test (SAT) is the formal handover document produced when capital equipment
is installed at a customer facility. The engineer runs a defined set of parametric tests,
records measured values against specification limits, and issues a verdict ‚Äî Pass, Conditional,
or Fail ‚Äî along with a signed acceptance statement. If deviations exist, the report records
them with severity, required actions, owners, and due dates.

SAT reports are contractual: they define the conditions under which equipment is accepted for
production operation and, in Conditional or Fail cases, form the basis of the supplier's
remediation obligation.

### Portfolio rationale

The SQR (Supplier Qualification Report) proved the Provenance engine works. But SQR is
procurement-facing: it assesses a supplier's organisational capability through a scored audit.
SAT is engineering-facing: it assesses a specific machine installation through parametric test
data. These are fundamentally different strategy mixes.

The SAT template demonstrates:
- The engine is domain-agnostic (not SQR-specific)
- The SPLIT table pattern: extractor table ‚Üí computed column ‚Üí narrative_llm analytical paragraph
- The v1 image-annotation pattern: structured numerics + prose engineer observation ‚Üí LLM synthesis
- A simpler classifier chain: fail_count integer ‚Üí verdict (vs SQR's composite_score float)

## 2. Strategy mix

### Comparison vs SQR

| Pattern                    | SQR                                          | SAT                                         |
|----------------------------|----------------------------------------------|---------------------------------------------|
| Classifier input           | `composite_score` (float, weighted average)  | `fail_count` (integer count of failed tests)|
| Classifier output          | Approved / Conditional / Not Approved        | Pass / Conditional / Fail                   |
| Primary data table         | `scorecard_table` (criteria, score, weight)  | `test_results_table` (test_id, param, spec, measured) |
| Computed column            | `weighted_score` per criterion               | `within_spec` Pass/Fail per test row        |
| Analytical narrative input | Per-criterion rows + aggregate score         | Per-test rows + pass_count / fail_count     |
| Lookup blocks              | `approval_conditions_blocks[verdict]`        | `acceptance_conditions_blocks[verdict]`     |
| Secondary narrative        | `risk_narrative` from risk register table    | `measurement_narrative` from annotation fields |
| Image handling             | None                                         | v1: structured annotation; v2: upload + vision |
| Total narrative_llm fields | 6                                            | 5                                           |
| Total sections             | 6                                            | 6                                           |
| Approximate field count    | ~47                                          | ~42                                         |

### Why the SAT strategy mix is different

SQR's classifier input is a continuous float because audit quality is a weighted aggregate of
many subjective scores. SAT's classifier input is a discrete integer because test pass/fail is
binary ‚Äî a test either meets spec or it does not. This makes the SAT classifier simpler and
more auditable: the verdict derivation is not a numerical judgement call, it is a count.

The SPLIT table pattern is the same in both templates, but the direction differs. In SQR,
`weighted_score` is a calculated value that feeds the classifier. In SAT, `within_spec` is a
classification result that is *summarised* by a downstream `narrative_llm` field after the
classifier has already fired.

## 3. Section-by-section design rationale

### Section 1 ‚Äî Equipment & Site Header

Three deterministic fields establish the document identity:

- `report_header_line` (`template_fill`): Produces the document title line
  `SAT-{report_id} ¬∑ {equipment_model} ¬∑ {site_name}`. Mirrors SQR's `report_title` pattern.
- `equipment_details` (`extractor`): Passes 12 intake fields through verbatim. No LLM needed;
  the data is structured and must be reproduced exactly.
- `issue_date` (`calculator`, expression=`today`): Auto-set to generation date. Consistent
  with SQR's `issue_date` field; no user input required.
- `acceptance_scope` (`lookup`): Equipment-category-keyed paragraph stating what the test
  covers. Six entries (one per category enum value). Prevents the LLM from ever describing
  scope ‚Äî scope is fixed by the applicable procedure, not synthesised.

### Section 2 ‚Äî Test Conditions

- `test_conditions_table` (`extractor`): Passes the 5-column ambient conditions table verbatim.
- `conditions_narrative` (`narrative_llm`): 1‚Äì2 sentences contextualising ambient conditions.
  Low complexity field; max_tokens=100. The prompt includes explicit omission instruction
  ("Omit any field that shows '‚Äî'") to handle optional columns gracefully.

### Section 3 ‚Äî Performance Test Results

The SPLIT table pattern in full:

1. `test_results_table` (`extractor` with `computed_columns`): The engineer's test rows plus a
   computed `within_spec` column. The computed expression is a Python ternary:
   `'Pass' if (measured_value >= spec_min and measured_value <= spec_max) else 'Fail'`.
   This is deterministic ‚Äî no LLM involved.
2. `pass_count`, `fail_count`, `total_tests` (`calculator`): Count fields derived from
   the augmented table. `pass_count` uses `count(test_results_table, within_spec='Pass')`.
3. `effective_conditional_limit` (`calculator`): Resolves the optional intake field
   `conditional_fail_limit` with a default of 1 if blank. This makes the Conditional verdict
   threshold configurable per engagement without requiring a code change.
4. `test_verdict` (`classifier`): Three rules evaluated in order.
   `fail_count == 0` ‚Üí Pass; `fail_count <= effective_conditional_limit` ‚Üí Conditional;
   `fail_count > effective_conditional_limit` ‚Üí Fail.
5. `results_narrative` (`narrative_llm`): Analytical 2‚Äì3 sentence summary. The prompt feeds
   the first 5 test rows individually with test_id, parameter, measured value, spec range, and
   within_spec result. The LLM is instructed to name specific test IDs ‚Äî a groundedness
   constraint that prevents generic summarisation.

### Section 4 ‚Äî Measurement Analysis

The v1 image-annotation pattern:

- `measurement_data` (`extractor`): Five structured numeric fields from intake (type, headline
  value/unit, secondary value/unit). These are the quantitative anchors for the narrative.
- `engineer_note` (`extractor`): Single prose field ‚Äî the engineer's observation about what
  the measurement shows. Reproduced verbatim; not summarised.
- `figure_caption` (`template_fill`): Produces `Figure: {measurement_type} ‚Äî {value} {unit}`.
  Deterministic label for the placeholder figure slot.
- `measurement_narrative` (`narrative_llm`): 2‚Äì3 sentence synthesis. The prompt instructs the
  LLM to (a) state measurement type and headline value first, (b) incorporate the engineer's
  observation exactly, (c) include secondary value if present. Faithfulness constraint: "do not
  add interpretations beyond what is written." This ensures the narrative is grounded in
  engineer-provided content rather than LLM inference about what the measurement implies.

**Why no image upload in v1:** Image upload requires a new intake field type (`image`), a
Supabase Storage bucket, a presigned URL flow, and a vision model call in `narrative_llm`.
That is a substantial addition to the engine. In v1, the pattern is validated with structured
fields and prose annotation. The v2 design is in Section 12.

### Section 5 ‚Äî Deviations & Corrective Actions

- `deviation_count` (`calculator`): `len(deviations) if deviations else 0`. Guards against
  null deviations table (optional intake field).
- `acceptance_conditions` (`lookup`, conditional on `test_verdict != 'Pass'`): Fixed-text
  block stating conditional or fail acceptance conditions. Only rendered when relevant.
- `deviations_table` (`extractor`, conditional on `deviation_count > 0`): Passes the
  deviations table verbatim. Six columns including severity enum and due date.
- `deviation_summary` (`narrative_llm`, conditional on `deviation_count > 0`): 2‚Äì3 sentence
  summary naming each deviation by test_id and severity. Instruction: "Do not add root cause
  analysis beyond what is stated in the description field." Mirrors the faithfulness constraint
  in `measurement_narrative`.

Both conditional fields use the same `condition` string mechanism as SQR's conditional fields.
The orchestrator evaluates the condition against the execution context before routing.

### Section 6 ‚Äî Acceptance Statement

- `tests_to_close` (`calculator`): Alias for `fail_count`. Exists as a named, labelled field
  for the export renderer to reference in the acceptance statement table.
- `executive_summary` (`narrative_llm`): 2‚Äì3 sentence executive summary. The prompt feeds
  eight structured values (equipment, site, customer, standard, verdict, pass count, deviation
  count). Exemplars cover all three verdict paths. The most important field in the report ‚Äî
  this is the primary text a customer reads.
- `signatory_block` (`lookup`): Fixed-text signatory footer with evaluator name, issue date,
  template version, and report ID. Same pattern as SQR's `signatory_block`.

## 4. The SPLIT table pattern

The SPLIT table pattern appears in both SQR and SAT and is the engine's primary mechanism for
producing analytical narratives grounded in structured tabular data.

**Step 1 ‚Äî extractor with computed_columns:** The raw intake table is augmented with one or
more computed columns derived by evaluating a Python expression per row against the row's
field values. The result is a new, enriched table in the execution context.

**Step 2 ‚Äî calculator fields:** Aggregate summary fields (counts, sums) are derived from the
enriched table. These are deterministic and appear in the report as labelled structured fields.

**Step 3 ‚Äî classifier:** The aggregate counts feed a classifier that emits a categorical
verdict. The classifier rules are evaluated in declaration order; the first matching condition
wins.

**Step 4 ‚Äî narrative_llm:** The prompt receives both the row-level detail (individual test
results) and the aggregate summary (pass_count, fail_count, verdict). The LLM is instructed to
reference specific row values ‚Äî test IDs, measured values, spec limits ‚Äî which grounds the
narrative in the actual data and makes it checkable.

**SQR vs SAT:**

In SQR, the computed column is `weighted_score` (a numeric calculation), and the classifier
input is the aggregate `composite_score`. The computed column feeds upward into the classifier.

In SAT, the computed column is `within_spec` (a categorical Pass/Fail), and the classifier
input is `fail_count` (a count of Fail rows). The computed column feeds a count, which feeds
the classifier. The direction is the same; the intermediate type differs.

## 5. The image-annotation pattern (v1)

### Problem statement

SAT reports in the field include measurement figures ‚Äî uniformity maps, process trend charts,
vibration spectra. These are the primary evidence for the acceptance verdict. A pure text
system cannot reproduce them. But integrating image upload into the engine is a large change.

### v1 design (this template)

In v1, the engineer provides:
- A structured identification of the figure type (enum: Thickness uniformity map, etc.)
- The headline measurement value and unit (e.g. 2.3%, nm, mm/s)
- An optional secondary measurement value and unit
- A prose observation describing what the figure shows (1‚Äì3 sentences)

The engine produces:
- A `figure_caption` via `template_fill`: `Figure: {type} ‚Äî {value} {unit}`
- A `measurement_narrative` via `narrative_llm`: 2‚Äì3 sentences anchored to the structured
  values and the engineer's own observation

The figure slot is a placeholder in the PDF/DOCX output. The narrative reads naturally as a
caption for a figure that would be attached separately.

### Groundedness constraint

The prompt includes: "Incorporate the engineer's observation exactly as stated ‚Äî do not add
interpretations beyond what is written." This is the same constraint applied in SQR's
risk_narrative v1.1 prompt, which improved groundedness score from 2.2 to 4.93. The engineer's
observation is the truth anchor; the LLM's role is synthesis, not inference.

### v2 design (future)

v2 adds a native `image` field type to the intake schema:

```yaml
measurement_figure:
  type: image
  required: false
  label: "Key measurement figure (PNG or JPEG, max 5 MB)"
  storage_bucket: sat-figures
  max_size_mb: 5
  accepted_formats: [image/png, image/jpeg]
```

The image field would:
1. Render a `react-dropzone` upload component in the intake wizard
2. Upload to Supabase Storage on drop (presigned URL, direct client upload)
3. Store the storage object key in `intake_data.measurement_figure`
4. At generation time, `narrative_llm` retrieves the presigned download URL and passes the
   image to the Claude vision API as a base64 image block alongside the text prompt

The `image_narrative` strategy would replace `narrative_llm` for vision-capable fields,
with the image content injected into the messages array as:
```python
{"type": "image", "source": {"type": "base64", "media_type": "image/png", "data": b64}}
```

The engineer observation field would remain ‚Äî it becomes grounding context alongside the image,
not a replacement for it.

## 6. Intake form walkthrough

### Step groupings (proposed for v2 intake wizard)

The SAT intake has more distinct conceptual groups than SQR:

**Step 1 ‚Äî Equipment & Site**  
equipment_model, equipment_serial, equipment_category, site_name, site_location, customer_name,
customer_contact_name, customer_contact_title, installation_date, test_date, evaluator_name,
acceptance_standard  
*12 fields; all identification; no conditionals*

**Step 2 ‚Äî Test Conditions**  
test_conditions (table: 5 columns, single required row)  
*1 table; 5 cell types; 2 required*

**Step 3 ‚Äî Performance Test Results**  
test_results (table: 7 columns, min 3 rows), conditional_fail_limit  
*1 table + 1 optional override; min 3 rows enforced by YAML min_rows*

**Step 4 ‚Äî Measurement Analysis**  
measurement_type, measurement_headline_value, measurement_headline_unit,
measurement_secondary_value, measurement_secondary_unit, engineer_observation  
*6 fields; 2 optional; engineer_observation is the longest text input*

**Step 5 ‚Äî Deviations**  
deviations (table: 6 columns, optional rows)  
*1 optional table; shown unconditionally but labelled "complete for failed tests"*

### Validation notes

- `spec_min` and `spec_max` columns in test_results are both `required: true` in the column
  schema. The frontend table component must enforce per-cell required constraints.
- `within_spec` is a computed column ‚Äî it must not appear as an editable cell in the intake
  table. Only `test_id`, `parameter`, `spec_min`, `spec_max`, `measured_value`, `unit`, `notes`
  are intake-time fields.
- The `deviations` table has no `min_rows` ‚Äî it is legal to submit with zero deviation rows.
  The validation rule `deviation_verdict_alignment` fires a warning (not error) if the verdict
  is Fail and no deviations were entered.

## 7. Generation pipeline

### Orchestrator execution order

The orchestrator processes sections in `order:` sequence and fields within each section in
declaration order. Key dependencies:

1. Section 1 fields (`equipment_details`, `issue_date`, `acceptance_scope`) ‚Äî no dependencies;
   execute immediately
2. Section 2 ‚Äî `test_conditions_table` must resolve before `conditions_narrative` can render
   its prompt interpolations
3. Section 3 ‚Äî execution order within the section is critical:
   - `test_results_table` (extractor + computed_columns) ‚Üí augments context
   - `pass_count`, `fail_count`, `total_tests` (calculators) ‚Üí depend on augmented table
   - `effective_conditional_limit` (calculator) ‚Üí depends on intake field
   - `test_verdict` (classifier) ‚Üí depends on `fail_count` and `effective_conditional_limit`
   - `results_narrative` (narrative_llm) ‚Üí depends on all of the above
4. Section 4 ‚Äî `measurement_data` and `engineer_note` must resolve before `measurement_narrative`
5. Section 5 ‚Äî `deviation_count` must resolve before `acceptance_conditions` condition evaluates
   and before `deviations_table` condition evaluates; `test_verdict` from Section 3 must be in
   context for `acceptance_conditions` source lookup
6. Section 6 ‚Äî `executive_summary` requires `test_verdict`, `pass_count`, `total_tests`,
   `deviation_count` all in context

### Deterministic vs LLM fields

| Strategy       | Fields                                                                         | LLM? |
|----------------|--------------------------------------------------------------------------------|------|
| template_fill  | report_header_line, figure_caption                                             | No   |
| extractor      | equipment_details, test_conditions_table, test_results_table, measurement_data, engineer_note, deviations_table | No |
| calculator     | issue_date, pass_count, fail_count, total_tests, effective_conditional_limit, deviation_count, tests_to_close | No |
| classifier     | test_verdict                                                                   | No   |
| lookup         | acceptance_scope, acceptance_conditions, signatory_block                       | No   |
| narrative_llm  | conditions_narrative, results_narrative, measurement_narrative, deviation_summary, executive_summary | Yes |

**27 deterministic fields, 5 LLM fields.** LLM generation is isolated; deterministic fields
complete synchronously during orchestration.

### Expected timing

Based on SQR generation timing (mean ~8‚Äì12 s for 6 LLM fields):  
- 5 LLM fields at ~2‚Äì3 s each = ~10‚Äì15 s total generation time
- Deterministic fields execute in <100 ms combined
- Supabase Realtime events emitted after each field; progress bar increments in real time

## 8. Review UI

The existing three-pane review layout requires no code changes for SAT. The engine routes
fields to the same rendering components based on field `type` and `strategy`.

### Strategy badge colours (existing convention)

| Strategy      | Badge colour |
|---------------|--------------|
| narrative_llm | Blue         |
| classifier    | Purple       |
| calculator    | Green        |
| lookup        | Grey         |
| extractor     | Amber        |
| template_fill | Slate        |

### Validation tab

Four validation rules in `validation_rules` will render in the Validation tab:
- `spec_completeness` (error) ‚Äî likely passes on any complete intake
- `total_tests_minimum` (error) ‚Äî enforced by intake min_rows=3; should always pass
- `deviation_verdict_alignment` (warning) ‚Äî active check; relevant for Fail verdicts
- `conditional_needs_dates` (warning) ‚Äî active check; relevant for Conditional verdicts

The `/revalidate` endpoint (added in Module 7.1) works for any template, not just SQR; no code
changes needed.

### Conditional fields in the review UI

Two fields are conditional:
- `acceptance_conditions`: only rendered if `test_verdict != 'Pass'`
- `deviations_table`: only rendered if `deviation_count > 0`
- `deviation_summary`: only rendered if `deviation_count > 0`

Conditional fields that evaluate false produce no `report_fields` row. The review UI must
handle gaps in section content gracefully ‚Äî already true for SQR's conditional fields.

- `type: table` fields ‚Äî use existing table HTML template (already used for SQR's scorecard and risk tables)

## 10. Eval framework ‚Äî synthetic test case designs

Seven cases covering all verdict paths, edge conditions, and the validation rules.

### Case 1 ‚Äî Clean Pass, Deposition system
- 5 tests, all Pass. `conditional_fail_limit` not set.
- Expected: `test_verdict = Pass`, `fail_count = 0`, `deviation_count = 0`
- Validation rules: all pass
- Narrative check: `results_narrative` must reference at least one test_id with measured value

### Case 2 ‚Äî Conditional, 1 failure, default limit
- 5 tests, 1 Fail. `conditional_fail_limit` not set (default 1).
- Expected: `test_verdict = Conditional`, `fail_count = 1`, `deviation_count >= 1`
- Validation rule `deviation_verdict_alignment`: passes (deviations entered)
- Validation rule `conditional_needs_dates`: passes only if deviation has due_date

### Case 3 ‚Äî Conditional, 1 failure, due date missing
- Same as Case 2 but deviation has no `due_date`.
- Expected: `test_verdict = Conditional`
- Validation rule `conditional_needs_dates`: **warning fires**

### Case 4 ‚Äî Fail, 2 failures, limit = 1
- 5 tests, 2 Fail. `conditional_fail_limit = 1`.
- Expected: `test_verdict = Fail`, `fail_count = 2`
- Validation rule `deviation_verdict_alignment` if deviations empty: **warning fires**

### Case 5 ‚Äî Fail, no deviations entered
- 3 tests, 2 Fail. `deviations` table is empty.
- Expected: `deviation_verdict_alignment` warning fires
- `deviation_summary` and `deviations_table` must not render (condition evaluates false)

### Case 6 ‚Äî Minimum test count (exactly 3)
- 3 tests, all Pass.
- Expected: `total_tests_minimum` rule passes (len=3 >= 3)
- Edge case: `results_narrative` prompt references rows 4 and 5 as "‚Äî"; LLM must omit them

### Case 7 ‚Äî Optional fields absent
- `measurement_secondary_value` and `measurement_secondary_unit` are blank.
- `customer_contact_name` and `customer_contact_title` are blank.
- `conditional_fail_limit` is blank.
- Expected: `effective_conditional_limit` defaults to 1; `conditions_narrative` omits blank
  ambient fields; `measurement_narrative` omits secondary value sentence

## 11. Prompt iteration notes

### Fields ranked by expected groundedness challenge (hardest first)

1. **`results_narrative`** ‚Äî highest complexity. References up to 5 individual test rows with
   test_id, parameter, measured value, spec range, and within_spec. The LLM must stay faithful
   to specific numeric values. Risk: LLM paraphrases or averages across rows. Mitigation:
   explicit instruction to "name any failed tests specifically with their measured vs spec
   values." Exemplar covers both all-pass and single-failure cases.

2. **`deviation_summary`** ‚Äî references 3 deviation rows. Risk: LLM adds root cause analysis
   not present in the description field. Mitigation: explicit constraint "Do not add root cause
   analysis beyond what is stated in the description field." The SQR `car_summary` v1.1 prompt
   (which achieved groundedness=5.0) is the direct model for this field.

3. **`executive_summary`** ‚Äî synthesis field that draws on 8 input values. Risk: verdict
   statement is ambiguous or hedged. Mitigation: "State the verdict clearly in the first
   sentence." Exemplar shows all three verdict paths.

4. **`measurement_narrative`** ‚Äî 3 inputs: type, structured values, engineer observation.
   Risk: LLM adds physical interpretation beyond what the engineer wrote. Mitigation:
   faithfulness constraint ("do not add interpretations beyond what is written").

5. **`conditions_narrative`** ‚Äî lowest complexity. 5 numeric inputs, single check against
   known thresholds. Risk: LLM invents acceptable ranges. Mitigation: ranges stated explicitly
   in prompt (temp 15‚Äì30¬∞C, humidity 20‚Äì70%).

### v1.1 prompt iteration candidates

If baseline eval shows groundedness below 3.5 for `results_narrative`:
- Feed all test rows as a bullet list rather than numbered lines
- Add per-row conditional: only include the spec range for rows where within_spec is Fail
- Tighten the "Do not infer causes of failure" constraint

If `executive_summary` below 3.5:
- Add explicit instruction: "Begin the first sentence with '[equipment_model] has [verdict]'"
- Remove exemplar path for verdict types not present in the test case

## 12. Image upload v2 design

### New intake field type: `image`

```yaml
measurement_figure:
  type: image
  required: false
  label: "Key measurement figure (PNG or JPEG, max 5 MB)"
  storage_bucket: sat-figures
  max_size_mb: 5
  accepted_formats: [image/png, image/jpeg]
```

### Frontend: react-dropzone component

- New `ImageField` component in `frontend/src/components/intake/ImageField.tsx`
- Renders a drop zone with file type validation and size check
- On drop: calls `supabase.storage.from('sat-figures').upload(path, file)`
- Stores the returned storage path in the form state for that field
- Preview thumbnail rendered below the drop zone

### Supabase Storage

- New bucket: `sat-figures` (private, no public access)
- RLS policy: `owner = auth.uid()` on object path prefix `{user_id}/`
- Presigned URL generated at generation time by the backend (1-hour TTL)

### Backend: image retrieval in narrative_llm

In `narrative_llm.py`, if the field's prompt references `{{intake.measurement_figure}}`:
1. Retrieve the storage path from `intake_data`
2. Call `supabase.storage.from('sat-figures').create_signed_url(path, 3600)`
3. Fetch image bytes via httpx
4. Encode as base64
5. Inject into the Claude messages array as an image block alongside the text prompt

### New strategy: `image_narrative`

A dedicated strategy for vision-capable fields avoids changing the existing `narrative_llm`
strategy. `image_narrative` has the same prompt/exemplar/max_tokens schema but adds:
- `image_source: str` ‚Äî intake field ID containing the storage path
- Image block injected before the text prompt in the messages array

### Export rendering (v2)

In the PDF renderer, the figure placeholder box is replaced by an `<img>` tag sourced from
the presigned URL. The Playwright renderer fetches the image at render time.

In the DOCX renderer, `python-docx`'s `add_picture()` method inserts the image bytes from a
downloaded temporary file.
