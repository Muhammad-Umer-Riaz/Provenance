# Module 3: Intake Form UI — Implementation Plan

## Context

Module 2 delivered the YAML template engine and all strategy handlers. Module 3 connects the template schema to the user — the YAML `intake:` block drives a multi-step wizard that collects structured input from the engineer. The YAML is the single source of truth: form fields, validation constraints, conditional visibility, and table column definitions all come directly from the template definition, not from hand-coded UI logic.

The key architectural claim being demonstrated: **schema-driven UI** — add a field to the YAML and it renders in the form automatically. This module also introduces the backend API surface (template serving + report creation) and the frontend API client.

---

## Decisions Confirmed

- **Form layout:** Multi-step wizard (5 steps), step-level validation before advancing
- **Validation:** `react-hook-form` + `zod` + `@hookform/resolvers` — Zod schemas generated from YAML field definitions
- **Table component:** `@tanstack/react-table` for all 3 intake tables
- **Post-submit flow:** Save intake only → redirect to `/reports` list (status = `draft`); generation is Module 4

## Wizard Step Breakdown

| Step | Title | Fields |
|------|-------|--------|
| 1 | Header Info | supplier_name, supplier_country, supplier_contact_name, supplier_contact_title, commodity_category, evaluator_name, duns_number |
| 2 | Qualification Context | qualification_type; conditional on non-Initial: previous_verdict, previous_composite_score, prev_otd_rate_pct, prev_defect_rate_pct, prev_invoice_accuracy_pct |
| 3 | Performance & Certifications | otd_rate_pct, defect_rate_pct, invoice_accuracy_pct, open_ncr_count, ncr_avg_close_days, certifications_held (multi_enum), SLA override fields (5 optional) |
| 4 | Audit Scorecard | audit_scores table (6 fixed rows: criterion read-only, weight editable, score 1–5 required, notes optional) |
| 5 | Risk & CARs + Review | risk_register table (add/remove rows), corrective_actions table (add/remove rows), final review summary + submit |

## Implementation Tasks

### Phase A — Backend API (do first; frontend depends on it)

**A1. Pydantic models** — add to `backend/app/routes/reports.py` (or a new `backend/app/schemas.py`):
```python
class ReportCreate(BaseModel):
    template_id: str       # "supplier-qualification-report"
    template_version: str  # "1.0"
    intake_data: dict[str, Any]

class ReportResponse(BaseModel):
    id: str
    template_id: str
    template_version: str
    status: str
    intake_data: dict[str, Any]
    created_at: str
```

**A2. `GET /api/templates/`** — new endpoint in a new `backend/app/routes/templates.py`:
- Call `get_loaded_templates()` from `loader.py`
- Return list with `template_id`, `version`, `name`, `description`, and full `intake` dict (the `IntakeFieldSchema` map)
- Frontend uses this to render the form and the templates browse page

**A3. `POST /api/reports/`** — add to `backend/app/routes/reports.py`:
- Accept `ReportCreate` body
- Resolve template from loaded templates (validate template_id + version exist)
- Insert row into Supabase `reports` table: `user_id` from `request.state.user`, `template_id`, `template_version`, `status='draft'`, `intake_data`
- Return `ReportResponse` (201)
- Use `get_supabase()` with service key for the insert (same pattern as health.py)

**A4. `GET /api/reports/`** — replace the stub:
- Query Supabase `reports` table filtered by `user_id`
- Return list of `ReportResponse`

**A5. Auth middleware** — verify that an auth middleware or dependency extracts the user from the Supabase JWT and attaches to `request.state.user`. If not already in place from Module 1, add a FastAPI `Depends` that calls `supabase.auth.get_user(token)`.

Register the new templates router in `backend/app/main.py`.

### Phase C — Field Components

All field components live in `frontend/src/components/forms/fields/`. Each receives:
- `name: string` — react-hook-form field name
- `schema: IntakeFieldSchema` — the YAML field definition
- `control: Control` — from parent `useForm`

**C1. `StringField.tsx`** — shadcn `Input`, registers with `control`  
**C2. `NumberField.tsx`** — `Input type="number"` with `min`/`max` from `schema.validation`  
**C3. `IntegerField.tsx`** — same as Number but `step={1}`  
**C4. `EnumField.tsx`** — shadcn `Select` with `schema.values` as options  
**C5. `MultiEnumField.tsx`** — shadcn `Checkbox` group; selected values rendered as pill badges  
**C6. `DateField.tsx`** — `Input type="date"`  
**C7. `ConditionalField.tsx`** — wrapper that evaluates the `schema.condition` string against current form watch values; renders children or null

For condition evaluation: implement a minimal evaluator that handles the exact patterns used in the SQR YAML:
- `qualification_type != 'Initial qualification'`
- These are simple equality/inequality comparisons — no need for a full expression engine

### Phase D — Table Components

All table components use `useFieldArray` from react-hook-form (registered at the wizard level) and `@tanstack/react-table` for rendering.

**D1. `AuditScorecardTable.tsx`**  
- 6 fixed rows from `schema.default_rows` (populated at form init, not removable)  
- `criterion` column: read-only, displays the criterion label  
- `weight` column: number input (0–1), pre-filled, editable  
- `score` column: integer input (1–5), required  
- `notes` column: string input, optional  

**D2. `RiskRegisterTable.tsx`**  
- Variable rows (min 1), add/remove buttons  
- Columns: `risk_item` (string, required), `likelihood` (int 1–5), `impact` (int 1–5), `owner` (string, required), `mitigation` (string, optional)  
- Display-only "Priority" column = `likelihood × impact`, sorted descending  

**D3. `CARTable.tsx`**  
- Variable rows (0 or more), add/remove buttons  
- Columns: `car_id` (string), `action_item` (string, required), `owner` (string, required), `due_date` (date), `status` (enum: Open / In progress / Closed / Overdue via shadcn Select)

Consider a **shared `EditableTable.tsx`** component that accepts column config and drives all 3 tables to avoid duplication.

### Phase E — Zod Schema Builder

**`frontend/src/lib/buildIntakeSchema.ts`** — converts a `Record<string, IntakeFieldSchema>` to a `z.ZodObject`:

```typescript
export function buildStepSchema(fields: Record<string, IntakeFieldSchema>, fieldNames: string[]): z.ZodObject<any> {
  // For each fieldName, map schema.type → z validator:
  // "string" → z.string().min(schema.required ? 1 : 0)
  // "number" → z.number().min(schema.validation?.min).max(schema.validation?.max)
  // "integer" → z.number().int()
  // "enum" → z.enum(schema.values)
  // "multi_enum" → z.array(z.string())
  // "table" → z.array(z.object(...from columns))
  // Wrap with .optional() if !schema.required
}
```

Each wizard step passes its own list of `fieldNames` to get a step-scoped schema.

### Phase F — Wizard

**F1. `frontend/src/components/forms/IntakeWizard.tsx`** — main wizard wrapper:
- Accepts `template: TemplateListItem`
- Initialises a single `useForm` instance (all fields registered here)
- Initialises `useFieldArray` for `audit_scores`, `risk_register`, `corrective_actions`
- Manages `currentStep` (0–4) state
- "Next" calls `form.trigger(stepFieldNames)` — only validates current step fields
- "Back" navigates without validation
- Step indicator bar at top (5 dots/labels)
- On step 5 "Submit", calls `createReport()` then redirects to `/reports`

**F2. Five step components** (`Step1HeaderInfo.tsx` … `Step5RiskAndCARs.tsx`):
- Each receives `control`, `watch`, `errors` props
- Each renders the relevant field components for its fields
- Step 5 includes a read-only summary section above the tables and the "Submit Report" button
