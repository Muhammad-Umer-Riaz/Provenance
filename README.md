# Provenance

**A schema-driven, field-level report generation engine with human-in-the-loop review.**

**[→ Live demo](https://d4q93waqw2e37.cloudfront.net)**

Built after watching technical reports get rewritten from scratch in industrial quality and acceptance-test workflows, where the underlying template never changes but every author writes it differently — and where an unverifiable paragraph generated by an LLM is a liability, not a productivity win.

Provenance takes a YAML template (defining sections, fields, and per-field generation strategies) plus structured human input and produces a draft report that a human reviews field-by-field before export. The LLM writes only the ~25% of the report that genuinely needs prose; the rest is deterministic by design. Every field carries its strategy, its inputs, and its full audit trail.

---

## The problem

Structured technical reports — supplier qualifications, site acceptance tests, non-conformance reports, incident postmortems — get written from scratch every time, despite having a stable underlying template. Engineers spend their judgment on formatting and prose rather than analysis. Output drifts across authors. The LLM tools that try to help either generate one big document from one big prompt (no provenance per field, no way to trust a paragraph) or lock the template behind a no-code editor (impossible to version, hard to audit). In regulated and engineering-driven workflows, that is a step sideways at best.

Provenance treats the report as a graph of fields, each with its own generation strategy, each independently traceable. Lookups stay deterministic. Calculators are pure functions. The LLM is invoked only where prose-from-structured-input genuinely adds value — and even then, its output is anchored to specific numeric inputs and gated by a human review step before anything leaves the system.

## What this is

- A **field-level orchestration engine.** Each field declares one of 7 generation strategies; the orchestrator routes accordingly.
- **YAML-first.** Templates are checked-in, versioned, and human-readable. The intake form, the validation rules, and the generation logic all derive from the same file.
- **Human-in-the-loop.** A 3-pane review UI surfaces every field with its strategy, inputs, and audit trail. Reports can't be exported until every required field is in `approved` status.

## What this isn't

- A chat interface. There is no conversation; there is a form, a generation step, and a review.
- A ChatGPT wrapper. Most fields never call an LLM. The ones that do return validated JSON, not free prose.
- A document scraper. Provenance assumes the input arrives as structured human entry, not as a Slack timeline or PagerDuty export.

## Architecture

### Field-level orchestration

A traditional LLM document-generation pipeline sends one big prompt and gets one big output. Provenance runs N independent field operations, each routed to exactly one strategy, each validated against a JSON schema, each retried independently on failure. One field failure does not abort the report.

```
            YAML template          structured human input
                  │                          │
                  └────────────┬─────────────┘
                               ▼
                       orchestrator.py
                               │
            ┌──────────────────┼──────────────────┐
            ▼                  ▼                  ▼
        lookup           classifier         narrative_llm
       extractor         calculator         direct_input
                       template_fill
                               │
                               ▼
                   per-field audit_log row
                               │
                               ▼
                  validator (cross-field rules)
                               │
                               ▼
                   Review UI (3-pane, field-level)
                               │
                               ▼
                  approved → export (PDF / DOCX / JSON)
```

### Strategy taxonomy

Every field in every template declares exactly one strategy. The orchestrator reads the strategy and routes accordingly.

| Strategy | What it does | Calls LLM? | Where used |
|---|---|---|---|
| `lookup` | Selects a fixed text block from a dictionary keyed on intake fields | No | Approval-conditions blocks (per verdict); acceptance-conditions blocks |
| `extractor` | Passes a user-entered value or table through verbatim (with optional computed columns) | No | All structured tables — scorecard, risk register, CAR table, test results |
| `calculator` | Computes deterministically from other fields using a safe expression evaluator | No | Composite score, weighted score per row, pass/fail counts, points to next tier |
| `template_fill` | Renders a fixed string with `{{field}}` interpolation | No | Report header line, simple structured prose |
| `direct_input` | Engineer types prose directly during intake; rendered as-is | No | Engineer-authored image captions, free-text remediation notes |
| `classifier` | Maps a value to an enum that gates downstream logic | Rule-based, no LLM in v1 | `qualification_verdict` from `composite_score`; `test_verdict` from `fail_count`; `risk_tier` from priority |
| `narrative_llm` | LLM writes a paragraph from structured field inputs, anchored to specific values | Yes | Scorecard summary, performance narrative, risk narrative, CAR summary, executive summary, recommendation |
| `grounded_llm` | LLM with retrieval against a reference corpus | Reserved for v2 | — |
| `hybrid` | User fills rows; LLM proposes additions in `review_required` mode | Reserved for v2 (user-fills only in v1) | — |

Two framing points:
1. `extractor` is named misleadingly. It does not extract from unstructured text. The intake form **is** the structured input; this strategy passes user input through with optional computed columns. No LLM.
2. Strategies are applied **per field**, not per section. A single section mixes a `narrative_llm` paragraph with `extractor` table cells and `lookup` boilerplate.

Three patterns appear across the two production templates. The first lives in SQR and demonstrates the core idea — a classifier gates an LLM narrative so the prose cannot diverge from the computed verdict. The second and third live in SAT and show how the same strategies compose into different shapes: a per-row aggregation pipeline, and a structured-numerics-plus-engineer-prose synthesis used for figure annotation. All three are pure YAML — no code change to support either template.

### Pattern: classifier gates the narrative

The most architecturally interesting pattern in the engine, and the one the SQR scorecard is built on. A `calculator` produces a composite score from the user's weighted inputs. A rule-based `classifier` maps that score to an enum verdict. A downstream `narrative_llm` field then writes prose anchored to specific scorecard rows *and* the verdict — both passed in as labelled prompt inputs. The model has no room to invent a score, a weight, or a verdict.

From `templates/supplier-qualification-report.yaml`:

```yaml
- id: composite_score
  type: field
  strategy: calculator
  expression: "sum(audit_scores[*].score * audit_scores[*].weight)"
  format: "0.00"

- id: qualification_verdict
  type: field
  strategy: classifier
  input: composite_score
  rules:
    - { condition: "composite_score >= 4.0", output: "Preferred" }
    - { condition: "composite_score >= 3.5", output: "Conditional" }
    - { condition: "composite_score >= 2.5", output: "Probationary" }
    - { condition: "composite_score < 2.5",  output: "Rejected" }

- id: scorecard_summary
  type: narrative
  strategy: narrative_llm
  prompt: |
    Composite score: {{composite_score}} / 5.00 — Verdict: {{qualification_verdict}}
    Audit scorecard — cite specific criterion names and their scores:
    1. {{scorecard_table[0].criterion}}: score {{scorecard_table[0].score}}/5, weight {{scorecard_table[0].weight}}
    ... (rows 2–6) ...
    Lowest: {{lowest_criterion_name}} at {{lowest_criterion_score}}/5
    Rules: Reference criterion names from the table above. Do not invent
    scores or weights not listed.
```

This is the difference between "generate a scorecard summary" and "summarize *these six rows* into a paragraph that mentions *this verdict*."

### Pattern: structured table → aggregator → narrative

SAT extends the chain to a different data shape. The engineer fills a structured **test_results** table directly during intake — one row per parametric test, with spec_min, spec_max, measured value, and unit. An `extractor` strategy then carries the table through with a **computed column** that flags each row Pass or Fail. Two `calculator` fields aggregate the rows into pass and fail counts. A `classifier` maps the counts to a verdict. A `narrative_llm` writes the analytical paragraph using all of the above — row IDs, measured values, counts, verdict — as labelled inputs.

From `templates/site-acceptance-test.yaml`:

```yaml
- id: test_results_table
  type: table
  strategy: extractor
  source: test_results
  computed_columns:
    - { name: within_spec,
        expression: "'Pass' if (measured_value >= spec_min and measured_value <= spec_max) else 'Fail'" }

- id: pass_count
  type: field
  strategy: calculator
  expression: "count(test_results_table, within_spec='Pass')"

- id: fail_count
  type: field
  strategy: calculator
  expression: "count(test_results_table, within_spec='Fail')"

- id: test_verdict
  type: field
  strategy: classifier
  input: fail_count
  rules:
    - { condition: "fail_count == 0",                           output: "Pass" }
    - { condition: "fail_count <= effective_conditional_limit", output: "Conditional" }
    - { condition: "fail_count > effective_conditional_limit",  output: "Fail" }

- id: results_narrative
  type: narrative
  strategy: narrative_llm
  prompt: |
    Overall: {{pass_count}} passed, {{fail_count}} failed of {{total_tests}} total. Verdict: {{test_verdict}}.
    Test results (reference specific test IDs and measured values):
    1. {{test_results_table[0].test_id}}: measured {{test_results_table[0].measured_value}}
       {{test_results_table[0].unit}} (spec {{test_results_table[0].spec_min}}–{{test_results_table[0].spec_max}})
       → {{test_results_table[0].within_spec}}
    ... (rows 2–5) ...
    Rules: Only reference test IDs and values from the table above. Do not infer causes of failure.
```

What SQR's scorecard pattern does for a weighted score, this pattern does for per-row pass/fail data. The architecture is identical — compute deterministically first, anchor the LLM to those values second — but the *shape of the input* is what the engineer types into a table, not a fixed set of weighted criteria. Adding new tabular intake to any template is a YAML edit, not a code change.

### Pattern: structured numerics + engineer prose → figure narrative

The third pattern is how SAT handles figures. Reports in acceptance-testing contexts include measurement charts and graphs (uniformity maps, process trend curves, vibration spectra). Provenance v1 deliberately does *not* feed the image itself to an LLM — multimodal interpretation of instrument plots is a v2 candidate. Instead, the engineer fills two structured inputs: a small set of **numeric headline values** describing what the figure shows, plus a free-text **`engineer_observation`** stating what they conclude from it. A `narrative_llm` field then synthesises both — the numbers come from the engineer's structured fields, and the interpretation comes from their prose. The model cannot invent a measurement value, and the engineer's interpretation is rendered verbatim or near-verbatim.

From `templates/site-acceptance-test.yaml`:

```yaml
- id: measurement_data
  type: structured_block
  strategy: extractor
  fields:
    - measurement_type
    - measurement_headline_value
    - measurement_headline_unit
    - measurement_secondary_value
    - measurement_secondary_unit

- id: engineer_note
  type: narrative
  strategy: extractor
  fields:
    - engineer_observation

- id: figure_caption
  type: structured_block
  strategy: template_fill
  template: "Figure: {{intake.measurement_type}} — {{intake.measurement_headline_value}} {{intake.measurement_headline_unit}}"

- id: measurement_narrative
  type: narrative
  strategy: narrative_llm
  prompt: |
    Write a 2–3 sentence technical paragraph describing the key measurement result.
    Measurement type: {{intake.measurement_type}}
    Headline value: {{intake.measurement_headline_value}} {{intake.measurement_headline_unit}}
    Secondary value: {{intake.measurement_secondary_value}} {{intake.measurement_secondary_unit}} (omit if blank or "—")
    Engineer observation: {{engineer_note}}

    State the measurement type and headline value in the first sentence.
    Incorporate the engineer's observation exactly as stated — do not add
    interpretations beyond what is written.
    Note: the figure itself is referenced but not reproduced here; the
    narrative is grounded in the structured values and engineer annotation above.
```

This is the design choice that protects the report from a real failure mode in measurement-heavy domains: an LLM looking at a colour-coded uniformity map and inventing a uniformity percentage. The image carries its own truth — the engineer can see if it's wrong — and the prose is constrained to what the structured fields and engineer's note say. v2 may add an opt-in multimodal strategy that reads the figure directly, but only with provenance for any vision-derived claim.

## Why this isn't a ChatGPT wrapper

Five architectural moves separate Provenance from a wrapper:

1. **Field-level orchestration, not document-level prompting.** N independent field operations, each routed to its own strategy, each validated, each retryable in isolation. Most fields never call an LLM.

2. **Structured output with JSON schema validation and per-field retries.** Every LLM call returns JSON matched against a per-field sub-schema; parse failures trigger up to three retries on the field, not on the whole document.

3. **Immutable audit log with full provenance.** Every field event (generation, regeneration, human edit) is an append-only row. The JSON export exposes the full chain — strategy, inputs, model, timestamp — for every field. This is the trust mechanism in regulated environments.

4. **Human-in-the-loop review UI with field-level diff.** 3-pane layout, strategy badges per field, inline edit with audit-log capture, per-field regenerate with optional natural-language guidance, approval gate enforced server-side before export.

5. **Eval framework measuring field-level accuracy and time-to-approval.** Deterministic fields exact-match tested. Narrative fields LLM-judged on groundedness (every claim traces to inputs) and factual consistency. Time-to-approval tracked per report. Prompt iteration is a versioned, measured engineering process — not a feeling.

- **No LLM framework** (no LangChain, no LlamaIndex). Raw OpenAI-compatible SDK calls via OpenRouter. Pydantic handles structured output validation directly.

- **Playwright over WeasyPrint** for PDF so the exported PDF is a pixel-perfect match to the review UI — no separate PDF template to drift against.

## Eval framework

Field-level evaluation mirrors field-level generation. Each strategy can be measured independently.

**Deterministic fields** (`lookup`, `extractor`, `calculator`, `classifier`, `template_fill`) are exact-match tested against an expected-values matrix. 15 synthetic SQR test cases cover every verdict outcome, every risk tier, every qualification type, and known edge cases (Conditional verdict with zero CARs, breach-level performance with non-Rejected verdict, overdue CARs on Probationary, etc.). Pass rate: 100% (105/105 fields).

**Narrative fields** (`narrative_llm`) are scored 1–5 by a separate judge model (Claude Haiku via OpenRouter, `temperature=0`) on two axes: groundedness (every claim in the paragraph traces back to a provided input) and factual consistency (text aligns with computed values). Threshold: mean groundedness ≥ 3.5.

**Baseline → v1.1 story.** First run produced mean groundedness 3.17 — failing. Two weak fields were named explicitly: `scorecard_summary` (2.00) and `risk_narrative` (2.20). Diagnosis pointed at vague prompt context — the model was being asked to "write a scorecard summary" without explicit per-row anchoring. Module 8 applied targeted fixes (per-criterion row anchoring on `scorecard_summary`; explicit likelihood × impact and mitigation-faithfulness constraint on `risk_narrative`; individual CAR row feeding on `car_summary`). Re-run produced mean groundedness **3.97** — passing, with `risk_narrative` jumping +2.73 and `car_summary` jumping +1.73.

Each change is documented per-field with before/after scores in `eval/PROMPT_HISTORY.md`. The full baseline is in `eval/BASELINE.md`.

**Time-to-approval** is the production success metric. On a sample SQR run: 45/47 fields approved on first pass (95.7%). The 2 non-first-pass fields were both in the weak-prompt cluster — consistent with the judge scores, which is itself a small validation of the eval design.

## Related work

A handful of tools exist in adjacent spaces. The architectural difference is worth naming because it determines fit.

**Vertical postmortem tools** — incident.io, Rootly, AlertOps "Agent Chronicle," ilert — pull context **automatically from upstream systems** (Slack timelines, PagerDuty events, Jira tickets, PR diffs). Their value proposition is "you don't have to type anything." That's powerful where the upstream context lives in machine-readable form.

**Quality / NCR workflow tools** — Process Street "Cora," Cassidy AI, Tulip "Quality Report Agent" — focus on workflow automation around the report (CAPA routing, approver chains, KPI rollups). Less emphasis on the generation logic, more on the process plumbing.

**Generalist document generation** — Templafy, Siav.AI, Fonic.ai — target enterprise Office 365 environments with locked-down templates and non-technical end users. Templates are managed centrally, not authored as code.

Provenance is structurally different on one axis: it assumes the context arrives as **structured human input**, not as a Slack scrape. That sounds like a weakness next to "fully automated," but it is exactly the right shape for environments where the input lives in instrument exports, engineer notebooks, or audit checklists rather than in chat logs. The schema-driven intake form **is** the value proposition for that audience. Regulated and engineering-driven workflows — supplier qualification, capital equipment acceptance, materials testing, clinical trial summaries — fall in this category, and the automated tools struggle there because there is no Slack channel to scrape for an XPS measurement run.

The other axis is **field-level orchestration with provenance per field.** None of the tools above expose the generation strategy per field, the audit trail per field, or the eval framework per strategy. Most of them generate at the document level and let the user edit the result. Provenance treats the field as the unit of generation, review, and evaluation — which is what makes the audit log defensible and the eval framework meaningful.

## Run it locally

**Prerequisites:** Python 3.11+, Node 20+, a Supabase project, an OpenRouter API key.

```bash
# Backend
cd backend
python -m venv venv
source venv/bin/activate                # macOS / Linux
# source venv/Scripts/activate          # Windows
pip install -r requirements.txt
playwright install chromium             # for PDF export
cp .env.example .env                    # then fill in Supabase + OpenRouter keys

# Frontend
cd ../frontend
npm install
cp .env.example .env                    # Supabase URL + anon key

# Database schema
# Apply the migrations in supabase/migrations/ to your Supabase project (Postgres SQL editor).
# RLS policies are included.
```

Then from the repo root:

```bash
bash start.sh
```

Or run the services individually:

```bash
# Terminal 1 — backend on port 8080
cd backend && source venv/bin/activate && uvicorn app.main:app --reload

# Terminal 2 — frontend on port 5173
cd frontend && npm run dev
```

Health check: `GET http://localhost:8080/` should return `{"status": "ok"}`.
